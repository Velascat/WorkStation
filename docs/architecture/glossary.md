# Architecture Glossary

Short, operational definitions for terms used across the platform. When a term is
used in architecture documents, code comments, or config files, it means what is
written here — no more, no less.

---

## Core Terms

**lane**
One of the three named execution paths a task can be sent down:
`claude_cli`, `codex_cli`, or `aider_local`. A lane identifies the runtime
environment and billing model for a single execution unit.

**runner**
The process that actually edits files within a lane. The Claude Code CLI binary,
the Codex CLI binary, or Aider are the three runners. A runner starts, does work,
and exits. It does not make strategic decisions.

**router**
A component that inspects a request and directs it somewhere else based on rules.
In this stack, SwitchBoard acts as the router for lane selection. The term is
descriptive of the function, not a reference to any removed component.

**selector**
The specific decision within SwitchBoard that chooses a lane. The `LaneSelector`
evaluates canonical lane policy and returns the selected lane/backend pair.

**proposal**
A structured suggestion generated by ControlPlane that a specific piece of work
is worth doing. In the canonical contract, a proposal carries task identity,
execution intent, target repository context, execution constraints, validation
profile, branch policy, and related metadata. It may be routed and executed, but
it does not include execution-layer details such as workspace paths or task-branch
names.

**workflow harness**
A system that wraps an execution unit in a structured, multi-step process. Archon
is the workflow harness. It loads a YAML DAG, executes nodes in order (or in
parallel), enforces approval gates, and tracks run state in a database. The harness
does not decide what to work on — it only controls how execution proceeds once a task
is in flight.

**coding backend**
The component that performs the actual code edits inside a lane. kodo is the coding
backend. It spawns an agent session, drives it to completion, and writes structured
artifacts. kodo does not select lanes and does not decide what to work on.

**outer shell**
An optional operator-facing runtime that sits above ControlPlane. OpenClaw is the
outer shell. It provides a human control surface but is not required for the system
to function autonomously.

## Lane Terms

**premium lane**
A lane that uses a paid CLI subscription for execution: `claude_cli` or `codex_cli`.
No per-call API keys are involved. Auth is handled by the CLI's own OAuth session.

**cheap lane**
The `aider_local` lane. Runs against locally deployed tiny models provided by
WorkStation. No external API calls. Zero marginal cost per run.

**local tiny-model lane**
Synonym for `aider_local`. Emphasises that the models are deployed on the local
machine by WorkStation, not accessed via a remote API.

## System-Flow Terms

**autonomy loop**
ControlPlane's four-stage cycle: `observe → analyze → decide → propose`. Runs
continuously (subject to budget and suppression guardrails) to generate a stream of
bounded, validated task proposals.

**lane selection**
The act of SwitchBoard evaluating a `TaskProposal` against policy and returning a
`LaneDecision` with the selected lane/backend and routing rationale. Lane selection
is policy-driven and happens once per task dispatch.

**execution dispatch**
The act of ControlPlane's execution boundary turning a routed task into an
`ExecutionRequest` and invoking the selected bounded adapter. Follows lane
selection.

**worktree isolation**
The practice of creating a separate git worktree for each task execution, so parallel
runs never share a working tree. Both kodo and Archon use worktree isolation.

**task board**
Plane. The canonical source of truth for task state, comments, and labels. ControlPlane
reads task state from and writes results back to the task board.

**artifact**
A structured output file written after a task execution. Examples: `diff_patch.txt`,
`validation.json`, `summary.json`, `outcome.json`. Artifacts are retained under
`tools/report/` and consumed by ControlPlane's feedback loops.

## Contract and Integration Terms

**TaskProposal**
The canonical proposal emitted by ControlPlane when it decides a task is worth
attempting. Carries proposal/task/project identity, task type, execution mode,
goal text, optional constraints text, target repository context, priority,
risk level, execution constraints, validation profile, branch policy, and
metadata such as proposer, labels, and proposal time. It does not contain
execution-layer details such as workspace paths or branch names resolved at run
time.

**LaneDecision**
The canonical routing decision emitted by SwitchBoard for a `TaskProposal`.
Records the proposal reference, selected lane, selected backend, routing
confidence, optional matched policy rule, optional rationale, optional
alternatives considered, and decision metadata such as timestamp and
SwitchBoard version. It references the proposal by ID and does not embed
execution logic.

**Backend**
An external execution system integrated through an adapter. kodo, Archon, and
OpenClaw are all backends from the platform's perspective. A backend is responsible
for running code; it is not responsible for deciding what to run or which lane to use.

**Adapter**
The boundary layer between the platform's canonical contracts and a specific backend's
wire format or API. Adapters translate inward (normalise backend responses into
platform types) and outward (translate platform requests into backend-specific calls).
The platform owns adapter contracts; backends do not define them.

**adapter-first**
The default integration posture for external systems. The platform integrates
through canonical contracts and adapters first, and only later evaluates whether
upstream modification is justified by retained evidence.

**integration friction**
A recurring limitation, workaround, or operational annoyance observed at an
adapter boundary. Friction becomes architecturally meaningful only when frequency,
severity, and impact justify more than local adaptation.

**upstream patch proposal**
A review-only recommendation that a narrow upstream change may now be worth
considering. It is not an implementation commitment and does not change active
runtime behavior by itself.

**ExecutionRequest**
A canonical, backend-neutral struct passed from ControlPlane's execution
boundary to a backend adapter after ControlPlane has combined a `TaskProposal`
with a `LaneDecision` and resolved execution-layer details. Contains the run,
proposal, and decision IDs; goal and optional constraints text; repository,
branch, and workspace details; allowed paths and execution limits; validation
commands; and request timestamp. Does not contain provider-specific fields.
Defined by the platform, not by the backend.

**ExecutionResult**
A canonical, backend-neutral struct returned by a backend adapter to
ControlPlane's execution boundary. Contains: outcome status, artifact paths,
validation results, and a brief summary. Defined by the platform, not by the
backend.

**ExecutionArtifact**
A file produced during or after execution and retained for downstream consumption.
Examples: `diff_patch.txt`, `validation.json`, `outcome.json`, `stderr.txt`.
ControlPlane reads artifacts to classify outcomes and drive the feedback loop.
Artifacts are backend-agnostic; the platform defines the expected schema.

**Telemetry**
Structured operational data emitted during system operation and retained for
analysis. Includes lane-selection decisions, execution outcomes, confidence
calibration events, budget usage, and heartbeat records. Telemetry feeds
ControlPlane's self-tuning regulator and operator dashboards.

**Contract-owned architecture**
The design principle that canonical data contracts (TaskProposal, ExecutionRequest,
ExecutionResult, LaneDecision) are defined and owned by the platform, not by any
external backend. Backends implement adapters that translate to and from these
contracts. This means the platform can swap backends without changing upstream
contract definitions.

---

## What These Terms Are Not

| Term used here | Is NOT the same as |
|----------------|--------------------|
| `lane` | An HTTP route path |
| `router` | A removed provider proxy or an HTTP route path |
| `selector` | A CSS selector or DOM concept |
| `proposal` | A pull request (proposals may eventually become PRs, but they are distinct) |
| `workflow harness` | The autonomy loop (that is ControlPlane's concern) |
| `coding backend` | The full AI system (it is one bounded component) |
| `outer shell` | A shell script or bash environment |
