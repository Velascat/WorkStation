# Architecture Glossary

Short, operational definitions for terms used across the platform. When a term is
used in architecture documents, code comments, or config files, it means what is
written here — no more, no less.

---

## Core Terms

**lane**
One of the three named execution paths a task can be sent down:
`claude_cli`, `codex_cli`, or `aider_local`. A lane identifies the runtime
environment and billing model for a single execution unit.

**runner**
The process that actually edits files within a lane. The Claude Code CLI binary,
the Codex CLI binary, or Aider are the three runners. A runner starts, does work,
and exits. It does not make strategic decisions.

**router**
A component that inspects a request and directs it somewhere else based on rules.
In this stack, SwitchBoard acts as the router for lane selection. The term is
descriptive of the function, not a reference to any removed component.

**selector**
The specific decision within SwitchBoard that chooses a lane. The `Selector` service
evaluates policy, scores eligible lanes, and returns the winner.

**proposal**
A structured suggestion generated by ControlPlane that a specific piece of work
is worth doing. A proposal includes a task description, confidence score, evidence
bundle, and candidate family label. Proposals may be executed automatically or
require human approval.

**workflow harness**
A system that wraps an execution unit in a structured, multi-step process. Archon
is the workflow harness. It loads a YAML DAG, executes nodes in order (or in
parallel), enforces approval gates, and tracks run state in a database. The harness
does not decide what to work on — it only controls how execution proceeds once a task
is in flight.

**coding backend**
The component that performs the actual code edits inside a lane. kodo is the coding
backend. It spawns an agent session, drives it to completion, and writes structured
artifacts. kodo does not select lanes and does not decide what to work on.

**outer shell**
An optional operator-facing runtime that sits above ControlPlane. OpenClaw is the
outer shell. It provides a human control surface but is not required for the system
to function autonomously.

## Lane Terms

**premium lane**
A lane that uses a paid CLI subscription for execution: `claude_cli` or `codex_cli`.
No per-call API keys are involved. Auth is handled by the CLI's own OAuth session.

**cheap lane**
The `aider_local` lane. Runs against locally deployed tiny models provided by
WorkStation. No external API calls. Zero marginal cost per run.

**local tiny-model lane**
Synonym for `aider_local`. Emphasises that the models are deployed on the local
machine by WorkStation, not accessed via a remote API.

## System-Flow Terms

**autonomy loop**
ControlPlane's four-stage cycle: `observe → analyze → decide → propose`. Runs
continuously (subject to budget and suppression guardrails) to generate a stream of
bounded, validated task proposals.

**lane selection**
The act of SwitchBoard evaluating a task request against policy and returning a
lane assignment. Lane selection is policy-driven and happens once per task dispatch.

**execution dispatch**
The act of sending a task to a specific lane runner (through kodo or directly). Follows
lane selection.

**worktree isolation**
The practice of creating a separate git worktree for each task execution, so parallel
runs never share a working tree. Both kodo and Archon use worktree isolation.

**task board**
Plane. The canonical source of truth for task state, comments, and labels. ControlPlane
reads task state from and writes results back to the task board.

**artifact**
A structured output file written after a task execution. Examples: `diff_patch.txt`,
`validation.json`, `summary.json`, `outcome.json`. Artifacts are retained under
`tools/report/` and consumed by ControlPlane's feedback loops.

---

## What These Terms Are Not

| Term used here | Is NOT the same as |
|----------------|--------------------|
| `lane` | An HTTP route path |
| `router` | 9router (removed) |
| `selector` | A CSS selector or DOM concept |
| `proposal` | A pull request (proposals may eventually become PRs, but they are distinct) |
| `workflow harness` | The autonomy loop (that is ControlPlane's concern) |
| `coding backend` | The full AI system (it is one bounded component) |
| `outer shell` | A shell script or bash environment |
